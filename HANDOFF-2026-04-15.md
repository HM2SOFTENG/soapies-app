## 5. Key Patterns & Conventions

### tRPC
- All endpoints in `server/routers.ts`
- Auth: `publicProcedure` / `protectedProcedure` / `adminProcedure`
- `ctx.user.id` = authenticated user's numeric DB id
- `ctx.user.role` = `'user'` or `'admin'`

### Database (Drizzle + MySQL)
- All queries in `server/db.ts`
- Two connection pools:
  - `getDb()` → Drizzle ORM (most queries)
  - `getRawPool()` → mysql2 pool (used for `member_signals` upsert/select)
- Schema in `drizzle/schema.ts` — do NOT edit the DB directly without a migration

### Community scoping (CRITICAL pattern)
When a user makes a request for community data, the server resolves their `communityId` from `profiles` and applies it automatically. Do NOT pass hardcoded community strings from the client. Example:
```ts
// In routers.ts — correct pattern:
const profile = await db.getProfileByUserId(ctx.user.id);
const communityId = (profile as any)?.communityId ?? undefined;
return db.getSomeData(communityId);
```

### Mobile navigation (expo-router)
- File-based routing — files in `mobile/app/` map to routes
- Bottom tabs in `mobile/app/(tabs)/`
- Admin routes in `mobile/app/admin/`
- Navigate: `router.push('/connections' as any)`

### Colors
```ts
// mobile/lib/colors.ts
```

## 8. Admin Workflow (for testing)

The full application lifecycle:
1. User submits application → status: `submitted`
2. Admin reviews in mobile admin → "Approve for Interview" → `applicationPhase: interview_scheduled`
3. User books an intro call slot at `/schedule-interview` (web)
4. Admin marks interview complete → `applicationPhase: interview_complete`
5. Admin selects member role (Member / Angel / Admin) → "Final Approve" → `applicationStatus: approved`
